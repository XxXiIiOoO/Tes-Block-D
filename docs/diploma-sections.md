# BlockTest: Text Sections For Diploma

## Novelty

Новизна разработанного решения заключается в объединении нескольких прикладных задач в одной учебно-практической платформе. В рамках проекта BlockTest создана единая система, которая сочетает автоматизацию тестирования блокчейн-приложений, выполнение сценариев в изолированной среде, централизованное хранение результатов и наличие кроссплатформенного клиента. В отличие от набора разрозненных скриптов, предложенное решение предоставляет целостную архитектуру, в которой пользователь управляет проектами, тестами и запусками через веб-интерфейс, а выполнение тестов осуществляется в отдельных временных Docker-контейнерах.

Практический элемент новизны также состоит в использовании изолированного выполнения для каждого запуска теста. Такой подход снижает влияние сторонних факторов между тестовыми сценариями, позволяет ограничивать ресурсы выполнения и сохраняет воспроизводимость результатов. Дополнительно в системе реализована очередь фоновых задач на базе Redis и TaskIQ, благодаря чему запуск тестов отделён от пользовательского интерфейса и не блокирует работу клиента.

Отдельным элементом новизны является мобильный режим работы клиента. BlockTest поддерживает адаптивный интерфейс и установку как PWA-приложение, что позволяет использовать единую систему как на настольных устройствах, так и на мобильных. При этом все клиенты работают с единым backend API и одной базой данных, что обеспечивает согласованность информации между устройствами.

## Practical Significance

Практическая значимость проекта BlockTest заключается в создании рабочего инструмента, который может применяться для организации и автоматизации тестирования прикладных блокчейн-решений в учебных, исследовательских и прототипных проектах. Платформа позволяет формализовать тестовые сценарии, запускать их в контролируемой изолированной среде, сохранять логи, результаты и историю выполнения, а также анализировать накопленную статистику по запускам.

Разработанная система может использоваться студентами, разработчиками и исследовательскими командами как основа для проверки сценариев, связанных с логикой смарт-контрактов, клиентских интеграций и вспомогательных командных проверок. За счёт использования контейнеров Docker каждый запуск изолирован от других, а значит повышается надёжность и воспроизводимость результатов тестирования.

Практическая ценность решения усиливается тем, что платформа реализована как полный fullstack MVP: в ней присутствуют аутентификация, управление проектами и тестами, фоновое выполнение задач, хранение истории запусков, экспорт результатов и аналитический дашборд. Это делает систему пригодной не только для демонстрации архитектурного подхода, но и для непосредственного использования в качестве рабочего прототипа.

## Architecture

Архитектура системы BlockTest построена по клиент-серверной модели и включает несколько взаимосвязанных компонентов. Клиентская часть реализована на React и Vite и отвечает за пользовательский интерфейс, маршрутизацию, авторизацию и визуализацию результатов. Серверная часть реализована на FastAPI и предоставляет REST API для работы с пользователями, проектами, тестами, запусками и статистикой.

Хранение данных организовано в PostgreSQL. В базе данных сохраняются пользователи, проекты, тестовые сценарии, запуски и журналы выполнения. Для контроля версий структуры базы данных добавлены миграции Alembic, что делает изменение схемы управляемым и воспроизводимым. Для фонового выполнения тестовых сценариев используется Redis и очередь задач TaskIQ. Это позволяет отделить обработку длительных операций от синхронного HTTP-взаимодействия.

Особое место в архитектуре занимает модуль изолированного запуска. Worker-процесс получает идентификатор запуска из очереди, создаёт временный Docker-контейнер, выполняет в нём команду теста, собирает stdout и stderr, сохраняет результаты и затем удаляет контейнер. Благодаря такому подходу обеспечиваются изоляция, ограничение ресурсов и контролируемый жизненный цикл каждого запуска.

С точки зрения взаимодействия компонентов архитектура работает следующим образом: пользователь создаёт тест через frontend, backend сохраняет конфигурацию сценария, затем по запросу на запуск backend создаёт запись запуска и ставит задачу в очередь. Worker выполняет тест в Docker-контейнере, сохраняет результат в PostgreSQL, а frontend получает обновлённые данные через REST API и отображает их на панели мониторинга.

## Mobile Client And Data Consistency

Мобильный клиент в проекте реализован не через отдельное нативное приложение, а через адаптивный web-клиент с поддержкой PWA. Такой подход позволяет использовать единую кодовую базу для настольных и мобильных устройств, уменьшает сложность сопровождения и при этом обеспечивает полноценный пользовательский сценарий на смартфоне или планшете. Пользователь может установить BlockTest на домашний экран мобильного устройства и работать с ним как с приложением.

Согласованность данных между устройствами обеспечивается за счёт единого backend API и общей базы данных PostgreSQL. Независимо от того, открыт ли сервис на компьютере, ноутбуке или телефоне, операции выполняются через одну серверную бизнес-логику и изменяют одну и ту же структуру данных. Это означает, что созданные проекты, тесты, запуски и журналы выполнения доступны на всех клиентах в одинаковом состоянии.

Дополнительно для повышения актуальности интерфейса реализован механизм автоматического обновления данных. Клиент выполняет повторную загрузку статистики и запусков при возврате пользователя в окно приложения и через интервальное обновление. Благодаря этому снижается вероятность расхождения между состоянием интерфейса и фактическим состоянием данных на сервере, что особенно важно для мобильных сценариев, где пользователь может часто переключаться между приложениями.

## Short Defense Version

В разработанном проекте BlockTest реализована платформа для автоматизации тестирования блокчейн-приложений в изолированной среде. Система включает backend на FastAPI, frontend на React, PostgreSQL для хранения данных, Redis и TaskIQ для фоновой обработки, а также Docker SDK для выполнения каждого теста во временном контейнере. Практическая значимость работы состоит в том, что результат представляет собой не набор макетов, а рабочий fullstack MVP, который поддерживает аутентификацию, управление проектами и тестами, историю запусков, логи, статистику и экспорт результатов. Дополнительным преимуществом является поддержка мобильного клиента в формате PWA и обеспечение согласованности данных между устройствами за счёт единого backend API и общей базы данных.

## Experiment Scenario

Для оценки эффективности платформы целесообразно провести серию из 10-20 повторяемых запусков EVM-пресетов в двух режимах: ручной запуск через Docker/командную строку и запуск через BlockTest. Для каждого запуска фиксируются длительность, статус завершения, наличие ошибок, полнота сохранения логов и возможность воспроизведения результата. Сравнение показывает, что BlockTest уменьшает число ручных операций, централизует хранение результатов, предоставляет экспорт JSON и ускоряет анализ ошибок за счёт диагностических сигналов.

Пример структуры таблицы для отчёта: номер запуска, название пресета, длительность ручного запуска, длительность запуска в BlockTest, статус, обнаруженные ошибки, наличие сохранённых логов и признак воспроизводимости. Такая таблица позволяет показать не только факт работоспособности приложения, но и практический эффект от автоматизации.

## Threat Model And Isolation Limits

Система рассматривает пользовательские команды и тестовые скрипты как потенциально недоверенную нагрузку. Для снижения риска каждый запуск выполняется во временном Docker-контейнере с ограничением памяти, CPU, количества процессов, тайм-аутом и управляемым сетевым доступом. Контейнер запускается с read-only root filesystem, без Linux capabilities, с параметром `no-new-privileges`, отдельным временным каталогом `/tmp` и проверкой Docker-образа по allowlist. После завершения контейнер удаляется, а результаты сохраняются в базе данных.

При этом Docker-контейнеры не являются полноценной заменой изолированной виртуальной машине. Worker имеет доступ к Docker socket, поэтому узел worker относится к доверенной инфраструктуре. Если для live-пресетов включается доступ к Docker Compose network, тестовый контейнер получает возможность обращаться к сервису Hardhat и другим доступным службам этой сети. Для промышленного развития системы рекомендуется запускать worker на отдельном изолированном узле или использовать rootless Docker, а доступ к Compose network оставлять только для контролируемых демонстрационных сценариев.

## Защита приложения

В backend реализованы меры защиты от распространённых веб-угроз. Для работы с базой данных используются SQLAlchemy-запросы с параметрами, поэтому пользовательский ввод не конкатенируется в SQL-строки. Это снижает риск SQL-инъекций в операциях поиска, фильтрации и выборки данных. Пароли пользователей хранятся в виде хэшей, а токены подтверждения почты и сброса пароля также сохраняются только в хэшированном виде.

Для защиты от brute force реализованы два уровня контроля: блокировка пользователя после серии неудачных попыток входа и rate limiting по IP-адресу. Лимитер использует Redis, а для локальной разработки и тестов предусмотрен in-memory fallback. Авторизация построена на короткоживущих access-токенах и refresh-токенах, которые сохраняются в базе в виде хэша, ротируются при обновлении и отзываются при logout или сбросе пароля.

Дополнительно backend проверяет длину и формат пользовательских полей, ограничивает допустимые Docker-образы через `DOCKER_ALLOWED_IMAGES`, добавляет Content Security Policy и при `BLOCKTEST_ENV=production` включает HSTS. В production-режиме приложение не стартует с небезопасными настройками: стандартным `JWT_SECRET_KEY`, wildcard CORS, demo-mode email, выдачей verification token в API-ответе, пустым allowlist Docker-образов или неограниченным сетевым доступом Docker-контейнеров.
